# 1. 第五类最终覆盖口径

以下内容是后续讨论中已经拍板的最终口径。若与 5.1–5.5 原文件中的早期说法冲突，以下内容优先。

## 1.1 时间结算最终口径

【最终覆盖口径】

- 走 1 格 = 1 天。
- 事件要么不额外加天数，要么额外 +1 天 / +整数天。
- 不存在 0.5 天、半天、小数天。
- “正常移动 +1 天”是基础移动时间，不是事件额外时间。
- 事件里写“+1 天”，意思是在走这一格已经消耗 1 天的基础上，额外再增加 1 天。

## 1.2 天气变化最终口径

【最终覆盖口径】

- 天气不再按“每走一格重新随机一个天气”处理。
- 天气按“天气段”生成。
- 当一段天气结束时，系统才按 5.1 的天气基础概率抽取下一段天气。
- 抽到天气后，先判断天气类别，再按该类别生成持续天数。
- 每天 / 每格仍然只结算当前天气段的有效天气，不会一天内叠多个天气。
- 天气段剩余天数每天减少 1；剩余天数为 0 后，下一天重新抽取新的天气段。

### 1.2.1 天气段持续时间表

| 天气类别 | 包含天气 | 持续时间 | 说明 |
|---|---|---:|---|
| 普通天气 | 晴朗、阴天等常规天气 | 5–7 天 | 普通天气应该形成稳定路段感，不要每天跳 |
| 特殊天气 | 小雨、暴雨、高温、酷热、寒潮、严寒、浓雾、重雾等非灰烬特殊天气 | 3–6 天 | 特殊天气有压力和机会，但持续时间不能过短，否则玩家无法围绕天气决策 |
| 灰烬风 | 灰烬风 | 4–5 天 | 灰烬风属于强环境压力，但不是最极端状态，可以持续数天形成灰烬路段 |
| 灰烬风暴 | 灰烬风暴 | 2–3 天 | 灰烬风暴持续最短，但风险最高；不建议拖太久压死玩家 |

### 1.2.2 天气段生成流程

| 步骤 | 规则 |
|---|---|
| 1 | 当前天气段剩余天数每天 -1 |
| 2 | 若剩余天数仍大于 0，继续使用当前天气 |
| 3 | 若剩余天数为 0，按 5.1 天气基础概率抽取下一段天气 |
| 4 | 根据抽到的天气类别，生成持续天数 |
| 5 | 写入 `currentWeather` 和 `weatherDaysRemaining` |
| 6 | 每天按当前天气结算补给、货物、战斗、外部事件权重、内部事件压力等 |

### 1.2.3 天气段例子

【原表例子 / 整理说明】

假设当前天气段为“暴雨”，持续 4 天。玩家每天走 1 格，则这 4 格都处于暴雨天气；第 4 天结束后，暴雨段归零，第 5 天开始前再按天气基础概率抽取下一段天气。

这样做的目的不是增加复杂度，而是让天气有“路段感”：玩家能围绕这几天的天气决定是否赶路、绕路、扎营、处理货物或照顾伤员。

## 1.3 货车耐久 0 与抛锚最终口径

【最终覆盖口径】

货车耐久 0 的处理以第 3 类“商队核心资源、补给、零件、货车、车厢、负重、士气、撤退与失败系统”中的抛锚系统为准。

- 普通远行中，货车耐久 0 = 抛锚，停在当前格修理。
- 货车耐久 0 不等于立刻失败，也不直接强制撤退。
- 抛锚后，商队不能继续移动，不能继续订单路线，也不能进行普通野外探索。
- 抛锚后，必须把货车耐久修到 20 以上，才能继续移动。
- 获得零件本身没用；有零件但不修车，仍然不能移动。
- 修到 10 也不能移动；必须修到 20 以上。
- 如果长期无法修复、补给崩溃、士气崩溃，才进入撤退风险或强制撤退流程。

### 1.3.1 抛锚定义

【原表内容】

| 货车耐久 | 状态 |
|---:|---|
| 21–100 | 可以正常移动 |
| 1–20 | 危险状态，但仍可移动 |
| 0 | 抛锚，不能移动 |

关键规则：

> 抛锚后，必须把货车耐久修到 20 以上，才能继续移动。

### 1.3.2 抛锚士气惩罚

【原表内容】

| 情况 | 士气变化 |
|---|---|
| 货车耐久第一次跌到 0 | morale -1 |
| 抛锚期间继续停留 | 不重复触发“跌到 0”的士气惩罚 |
| 修到 20 以上后再次跌到 0 | 再次 morale -1 |

说明：

- 抛锚是重大危机。
- 但同一次抛锚期间，不应该每天重复扣“跌到 0”的固定惩罚。
- 只有修到 20 以上、重新恢复移动能力后，再次跌到 0，才再次触发士气 -1。

### 1.3.3 抛锚期间限制

【原表内容】

| 项目 | 抛锚状态规则 |
|---|---|
| 正常移动 | 禁止 |
| 自动行进 | 禁止 |
| 继续订单路线 | 禁止 |
| 野外探索 | 禁止 |
| 扎营 | 等同被迫停留 |
| 修车 | 可以 |
| 使用零件修车 | 可以 |
| 无零件临时修车 | 可以 |
| 主动撤退 | 可以，但代价极大 |
| 内部事件 | 概率提高 |
| 外部事件 | 只可能触发非移动型外部事件 |

### 1.3.4 抛锚期间每日结算

【原表内容】

| 项目 | 效果 |
|---|---|
| 天数 | day +1 |
| 补给 | food -2，按被迫野外停留 / 扎营处理 |
| 恶劣环境 | food -3 |
| 订单期限 | -1 天 |
| 内部事件 | 概率大幅提高 |
| 伤员恢复 | 正常情况下可恢复；若触发伤员恶化则反向增加恢复时间 |
| 货物风险 | 如果货车刚受损，可能触发货物松动 |
| 士气 | 由抛锚事件、争吵事件、断粮事件等扣除 |

### 1.3.5 抛锚修复方式

【原表内容】

| 修复方式 | 消耗 | 恢复货车耐久 | 说明 |
|---|---|---:|---|
| 普通角色用普通零件修车 | 1 普通零件 | +10 | 快速但效率低 |
| 修补师用普通零件修车 | 1 普通零件 | +16 | 修补师职业价值 |
| 普通角色用精制零件修车 | 1 精制零件 | +16 | 更贵更强 |
| 修补师用精制零件修车 | 1 精制零件 | +24 | 长线救命 |
| 普通角色用旧文明零件修车 | 1 旧文明零件 | +24 | 极贵救命资源 |
| 修补师用旧文明零件修车 | 1 旧文明零件 | +36 | 强力修复 |
| 普通角色无零件临时修车 | 1 天 + food | +3 | 非常慢 |
| 修补师无零件临时修车 | 1 天 + food | +5 | 仍然慢 |
| 特殊事件修车 | 事件决定 | 不固定 | 村庄、路人、商队救援等 |

补充规则：

- 抛锚后最少要修到 20 以上。
- 如果没有零件，就只能靠无零件临时修车慢慢爬。
- 无零件临时修车会消耗天数和补给，因此会继续压订单期限和队伍士气。

### 1.3.6 抛锚期间内部事件方向

【原表内容】

抛锚期间的内部事件，重点不是各种机械惩罚，而是队伍人心和压力。

| 内部事件方向 | 说明 |
|---|---|
| 队员争吵 | 队员质疑路线、质疑是否应该继续 |
| 要求撤退 | 某个队员建议立刻撤退 |
| 伤员抱怨 | 重伤角色无法承受继续停留 |
| 补给分配争议 | 谁吃多少、是否给伤员更多 |
| 货物优先级争吵 | 是保订单货，还是抛弃货物保命 |
| 修补师压力 | 修补师被要求连夜抢修，可能影响状态 |
| 领队威信事件 | 玩家需要选择安抚、强压、承诺奖励、分配补给 |

设计目的：

> 抛锚应该让玩家感觉“商队快散了”，而不是只是几个数值扣除。

### 1.3.7 抛锚期间伤员恶化

【原表内容】

| 触发 | 效果 |
|---|---|
| 伤员恶化 | 当天不减少重伤恢复时间 |
| 额外惩罚 | 重伤剩余恢复时间 +1 天 |

【原表例子】

某角色重伤剩余恢复 3 天。当天触发伤员恶化后：

- 这一天不恢复；
- 还额外增加 1 天；
- 结果变成剩余恢复 4 天。

这比单纯扣生命更清楚，也更符合“伤势被拖坏了”的感觉。

### 1.3.8 货物松动触发条件

【原表内容】

货物松动不应随便触发。

最终规则：

> 货物松动只在货车耐久下降后才可能触发。

| 来源 | 是否可能触发货物松动 |
|---|---|
| 战斗中货车受损 | 是 |
| 路况导致货车受损 | 是 |
| 灰烬兽冲撞货车 | 是 |
| 跌落、塌方、泥坑导致车损 | 是 |
| 单纯扎营 | 否 |
| 单纯住宿 | 否 |
| 没有车损的普通移动 | 否 |

整理说明：

货物松动是“车被震坏、撞坏、颠坏”后的后续风险，不是凭空发生。

### 1.3.9 抛锚主动撤退

【原表内容】

如果车坏了、多人重伤、没有零件、补给不足，玩家需要一个“认输但不废档”的按钮。所以抛锚时也可以主动撤退，但代价必须非常重。

| 项目 | 规则 |
|---|---|
| 是否允许 | 允许 |
| 触发条件 | 抛锚状态下，玩家选择放弃货物撤退 |
| 货物损失 | 全部货物损失 |
| 士气结果 | morale 降到 3 |
| 货车结果 | caravanHp 恢复到 20 |
| 角色结果 | 保留当前伤势，不额外强制全员重伤 |
| 撤退目标 | 最近已探明安全点 |
| 安全点范围 | 城市、城镇、驿站 |
| 是否算失败 | 算严重失败，但不是废档 |

### 1.3.10 抛锚撤退与士气崩溃撤退区别

【原表内容】

| 类型 | 触发 | 货物损失 | 士气结果 | 货车结果 | 说明 |
|---|---|---:|---|---|---|
| 主动抛锚撤退 | 玩家选择 | 100% | 降到 3 | 恢复到 20 | 玩家主动止损 |
| 士气崩溃撤退 | morale = 0 | 50% | 恢复到 1 | 视情况恢复最低状态 | 队伍失控 |
| 普通主动撤退 | 玩家选择 | 默认不全损 | morale -1 | 正常结算 | 路线风险太高时回撤 |

说明：

- 主动抛锚撤退损失货物最严重，但保留较高士气。
- 士气崩溃撤退货物损失较少，但队伍状态更差。

# 2. 第五类总定位与总运行流程

【整理说明】

第五类不是单独的天气表，也不是单独的随机事件表。它由五个子模块组成：

| 子模块 | 名称 | 作用 |
|---|---|---|
| 5.1 | 天气系统 | 决定当前天气段与天气对远行、事件、战斗、货物、内部压力的影响 |
| 5.2 | 外部事件系统 | 决定玩家在野外格子上遇到什么外部世界事件 |
| 5.3 | 内部事件系统 | 决定商队内部是否因压力爆发争执、抱怨、恐惧、互助等事件 |
| 5.4 | 事件冷却系统 | 对刚发生过的同类事件降权，避免重复刷屏和连续连炸 |
| 5.5 | 时间结算系统 | 统一处理走格、停留、绕路、搜索、修理、休整、撤退等时间口径 |

第五类运行时，建议按以下顺序：

1. 玩家走 1 格，基础时间 +1 天。
2. 根据当前天气段，应用当天有效天气。
3. 根据当前自然区域，判断是否发生外部强事件。
4. 若发生强事件，从 E 类事件池抽取，但先应用 5.4 降权冷却。
5. 事件发生后，根据 5.5 时间规则和资源结算规则落账。
6. 外部事件写入压力标签，供 5.3 内部事件系统使用。
7. 若商队压力达到 5.3 的检查条件，再判断内部事件。
8. 内部事件同样写入 5.4 同类型降权，并按 5.5 结算。

# 5.1 天气系统

【原表内容】来源文件：`余烬商队_第5.1类_天气系统阶段报告.xlsx`。以下尽量保留原文件内容、数值、特殊情况和原表例子；若与第 1 章最终覆盖口径冲突，以第 1 章为准。


《余烬商队》第 5 类：天气系统阶段报告
天气、环境压力、风险收益与事件接口
一、天气系统的定位
天气系统属于《余烬商队》第 5 类：
天气、环境、外部事件、内部事件、区域生态系统。
天气系统不是单独存在的装饰，也不是单纯给玩家扣资源的惩罚表。
它的核心作用是：
让商队远行过程中的每一天都产生不同判断。
玩家看到天气后，需要判断：
今天适不适合赶路；
要不要扎营等天气过去；
要不要绕路去水源；
要不要冒险穿过灰烬区；
重伤队员能不能继续撑；
怕潮货、怕热货、灰烬货是否需要保护；
当前天气下是否适合狩猎、追踪、采集或探索。
天气系统最终要服务于游戏核心体验：
商队远行 + 风险管理 + 路线选择 + 资源消耗 + 跑商经营。
二、天气系统设计原则
2.1 天气不是纯惩罚
每一种特殊天气都必须同时包含：
内容	说明
压力	让玩家付出代价，例如补给压力、伏击风险、货损风险
机会	给玩家利用空间，例如水源、痕迹、猎物、冰面捷径、灰烬材料
选择	让玩家判断继续赶路、扎营、绕路、冒险还是放弃
【原表例子】
例子：
天气	压力	机会
暴雨	怕潮货风险、泥路、重伤恶化	雨水补给、雨后痕迹
酷热	额外补给消耗、重伤恶化	普通遇敌下降、水源价值提高
严寒	低温事件、重伤恶化	野兽迟滞、保鲜、冰面捷径
重雾	伏击率提高、迷路判断	猎物线索、异常线索
灰烬风暴	灰烬兽、污染、重伤恶化	灰烬材料、旧文明残骸、遗迹线索
2.2 天气主要改变“权重”，不直接强塞结果
天气不应该经常直接写成：
今天暴雨，所以你损失 3 个货物。
更合适的做法是：
暴雨提高“泥路事件”“货物受潮事件”“雨后痕迹事件”的出现概率。
也就是说，天气影响的是：
外部事件池；
内部事件池；
战斗环境；
货物风险；
路线判断；
灰烬污染扩散；
重伤队员恶化概率。
2.3 一天一个天气
天气按天结算。
基础口径：
一天只生成一个主要天气。
当天所有远行、事件、战斗、货物、伤员结算都受这个天气影响。
不做一日多次随机天气，避免结算混乱。

四、每日天气结算顺序
建议每日结算顺序如下：
顺序	内容
1	判断当天基础天气
2	应用区域、剧情或特殊路线修正
3	显示天气提示
4	应用天气对补给、货物、战斗、污染、伤员的基础影响
5	修改当天外部事件池和内部事件池权重
6	玩家选择移动、扎营、绕路、修车、净化或等待
7	结算当天事件、战斗、货损、车损、伤员状态
【原表例子】
例子：
今天是暴雨。
系统先提示“暴雨，火焰伤害降低，怕潮货风险提高，重伤恶化概率提高”。
然后当天外部事件池中“泥路、陷车、雨后痕迹、货物受潮”权重提高。
玩家可以选择继续赶路、扎营收集雨水、整理防潮货，或者绕开泥路。

6.3 火焰伤害降低
天气	火焰伤害倍率
正常天气	1
小雨	0.75
暴雨	0.5
规则：
下雨削弱所有火焰伤害，敌我双方都生效。
但下雨不削弱燃烧扩散。
设计原因：
如果雨天同时削火焰伤害和燃烧扩散，火焰构筑会在雨天完全不能玩。
所以只削伤害，不削扩散。
【原表例子】
例子：

燃烧状态仍然可以扩散，只是伤害变低。

防潮工具包括：
工具	效果例子
防潮储物柜	放入其中的怕潮货不会受潮

防潮包装	降低受潮风险
密封货箱	保护一组货物
规则口径：
下雨不自动毁货。

玩家可以通过防潮工具、整理货物、扎营避雨、减少怕潮货来管理风险。
【原表例子】
例子：
玩家带着药品订单遇到暴雨，系统可能触发“药箱受潮”事件；
如果药品放在防潮箱里，则可以避免或降低风险；
如果玩家选择继续赶路，可能节省订单时间，但提高货损风险；
如果玩家扎营整理货物，会消耗一天，但降低受潮风险。

三、远行基础规则
《余烬商队》的远行采用：
一天一格。
一格就是一步。
一步就是一天。
玩家走到某个格子后，在当前格触发事件。
所有外部事件必须遵守以下口径：
行为	正确结算
快速查看 / 简单交谈 / 快速处理	不额外消耗天数
停留采集 / 修路 / 深入搜索 / 仔细侦察	当前格额外消耗 1 天
绕路	后续路线变长，仍按一天一格移动
走近路	后续路线变短，但风险提高
强行通过	当前格结算风险，如车损、货损、遇敌
标记位置	当前不深入处理，后续可能回来
禁止写：
半天；
半格；
减速 50%；
今天还能继续前进；
处理完再走一格；
直接传送到远处。
统一写法：
不额外消耗天数。
或者当前格额外消耗 1 天。
路线变长 / 变短表现为后续格子数量变化。

十一、E1 遇敌事件
11.1 定位
E1 是商队在远行过程中遭遇敌对力量。

十三、E3 资源事件
13.1 定位
E3 是发现资源机会，但不是白送资源。
核心口径：
快速采集，只拿一点。
想拿更多，就额外停留一天。
想拿好东西，就承担风险。
不想冒险，就放弃。
资源事件必须克制。
原因：
资源太多会破坏经济。
玩家如果走几步就白拿补给、零件、材料，跑商压力会消失。
13.2 资源类型

G2	狩猎 / 采集类	通过狩猎或采集获得	猎物、药草、干柴、兽骨
G3	零件 / 材料类	修车、交易、升级相关	备用零件、木材、黑石、金属残料
G4	废弃货物 / 可交易物	可出售、交付或临时使用	遗落货箱、旧商队货
G5	特殊 / 高风险资源	高价值但带风险	灰烬材料、污染残骸、旧时代样本

快速采集：

最终限制：
资源事件可以有诱惑，但不能让玩家稳定白拿资源。

二十、E10 路线判断事件
20.1 定位
E10 是：
玩家在当前格发现前方路线出现分歧、近路、绕路、危险路径、旧路、地形通道或剧情方向提示，需要决定下一步怎么走。
它不是道路事件，也不是痕迹事件。
事件	核心
E2 道路事件	路出了问题，玩家处理当前通行障碍
E9 痕迹 / 情报提示	看到线索，提示某个方向大概率有事
E10 路线判断事件	出现路线选择，玩家决定走哪条路
一句话：
E9 是“看见线索”。
E10 是“选择路线”。

二十一、E11 野外落脚点 / 临时营地事件
21.1 定位
E11 是：
玩家在野外发现一个可以临时停靠、避险、整理、观察或短暂休整的小地点。
它不是城市，不是驿站，也不是大型建筑。
它可以理解为：
野外已有的旧营地、天然庇护点、旧商队宿点、避风点、平整修车点。
21.2 E11 和普通扎营的区别
普通临时扎营可能有额外代价。
但 E11 是“发现可用落脚点”，所以它有优势。
项目	E11 处理
是否算新建营地	不算
是否触发“第一天建营地双倍消耗”	不触发
是否额外增加内部事件概率	不增加
事件本身是否额外消耗补给	不额外消耗
如果玩家选择额外停留 1 天	按正常时间 / 补给规则结算，不额外惩罚
是否展开内部事件	当前不展开，后续内部事件系统再处理
核心口径：
E11 是野外难得的安全喘息点。
它不该像普通扎营一样额外惩罚玩家。

二十二、E12 关卡 / 拦路 / 通行代价事件
22.1 定位
E12 是：
商队在路上遇到人为设置的通行阻碍、收费点、拦路者、临时路卡或盘查。
它不是普通 E1 遇敌。
它也不是 E4 商队 / 旅人。
E12 的重点是：
玩家先面对谈判和通行选择，而不是立刻战斗。
22.2 E12 不宜高频
关卡不能太随机。
原因：
荒郊野外突然频繁出现关卡会很怪。
关卡更适合固定点、半固定点、狭窄通道、桥口、峡口、商路节点、强盗控制区。
所以 E12 基础权重要低。
尤其纯野外区域，E12 不能高。
当前 E12 更多表现为：
土匪 / 强盗在前面收过路费，或者临时占据某个通行点。

二十三、E13 地标 / 视野 / 侦察事件
23.1 定位
E13 是：

E13 更像：

E13 可以揭示周围区域，但不能全开地图。

二十六、当前阶段总原则
26.1 外部事件不是越多越好
外部事件要让路上有事，但不能让玩家觉得每一步都被系统折磨。
核心口径：
路上经常有事，但不是经常打架，也不是经常白捡资源。
26.2 资源必须最克制
无论是 E3 资源事件、E6 天气资源，还是 E7 小型遗迹奖励，都必须遵守：
规则	说明
概率低	资源事件不能高频
给量少	快速获得通常只给 1 点级别收益

四、5.3 的核心原则
4.1 内部事件不是每日随机骚扰
内部事件不再设计成：
每天移动结束后都有固定概率触发。
改为：
只有商队出现明显压力时，才计算内部事件概率。
如果商队状态很好：
士气高；
补给足；
零件足；
车况好；
无重伤；
没有订单危机；
没有灰烬污染；
没有恶劣天气压力；
则内部事件概率可以是 0%。
4.2 内部事件是压力反馈
内部事件的作用不是惩罚玩家，而是反馈：
商队当前是否撑得住。
外部世界制造压力，内部事件表现压力。
例如：
外部或状态压力	内部反馈
补给不够到下个驿站	队员讨论是否减口粮或撤退
零件为 0	队员担心车坏了没人修
车况很差	队员要求停下修理
重伤员多	队伍要求休整或治疗
灰烬污染货物扩散	队员要求封存 / 净化
订单快没报酬	队伍争论是否继续
士气低	队员抱怨、恐慌或质疑路线
4.3 内部事件必须有解决手段
内部事件不能只是：
士气 -1；
补给 -1；
伤员恶化；
货损增加。
它必须让玩家有选择：
解决方式	例子
花补给	分发口粮、安抚队伍
花时间	停留 1 天修理 / 休整
花零件	处理车况问题
用装备	灰烬封存箱、防潮箱等
用角色能力	修补师处理伤员，护路人稳定队伍
改路线	去最近安全点、绕开危险区域
接受后果	不处理，内部压力继续积累
4.4 玩家管理得好，应降低内部事件概率
内部事件系统必须有减压项。
否则会变成折磨玩家。
可降低内部事件概率的状态包括：
士气高；
补给大量富足；
零件大量富足；
车况很好；
无重伤；
E11 良好落脚点；
已建成营地中躲避恶劣天气；
城市 / 驿站休整。
核心口径：
玩家管理得好，队伍就稳定。
玩家把商队拖到极限，内部事件才会爆发。

六、内部事件触发概率公式
内部事件采用：
压力项叠加制。
公式：

> 内部事件概率 = 士气修正 + 补给修正 + 零件修正 + 车况修正 + 重伤修正 + 天气修正 + 灰烬污染修正 + 订单 / 货损修正 + 扎营修正 + 安全缓冲。

最终限制：

| 项目 | 数值 |
|---|---:|
| 最低概率 | 0% |
| 最高概率 | 100% |
| 安全缓冲总上限 | -40% |
说明：
所有压力可以叠加。
所有减压也可以叠加。
但最终概率不能低于 0%，不能高于 100%。
减压项最多抵消 40%。

八、补给修正表
补给不按固定数量判断。
不能简单写：
补给少于 5 就加压。
原因：
如果玩家只是从城市去附近村庄，只需要 2–3 天，那么带 4 点补给是合理的。
但如果下一个驿站还有 20–40 格，带 4 点补给就是危险的。
所以补给压力按：
当前补给能否支撑到下一个安全点。
安全点包括：
城市；
驿站；
小镇；
可恢复据点；
特殊安全营地。
两驿站之间距离按：
20–40 格。
8.1 补给动态修正表
补给状态	判断方式	内部事件概率修正
大量富足	到达下个安全点后还剩很多补给	-0.1
少量富足	能到达下个安全点，且有正常余量	0
勉强够	正好够到达，几乎没有余量	0.1
略微不足	差一点才能到达安全点	0.2
明显不足	按当前路线明显无法到达安全点	0.3
补给为 0	不管距离多近，都进入饥饿压力	额外 +30%
说明：
少量富足是正常准备，不减压。
大量富足才说明队伍有底气，因此 -10%。
补给为 0 是额外饥饿压力，即使安全点很近，也会引发内部不安。

但有一个硬规则：
零件为 0，统一 +20%。
零件为 0 不再区分：
车况好不好；
路线安全不安全；
是否山地湿地；
是否已经抛锚。
车况问题由车况表单独处理。

十四、订单 / 货损修正表
订单压力接回原订单规则。
当前订单规则：
状态	结果
正常期限内完成	正常报酬
第一次超时后完成	原报酬的 50%
再次超时后完成	没有报酬
所以订单不是“一超时就失败”。
它是分阶段降低收益。

货损状态	内部事件概率修正	说明
货损 <30%	0	可接受
货损 30%–49%	0.1	队员开始担心收益
货损 50%–69%	0.2	明显争执
货损 ≥70%	0.3	接近订单失败 / 收益大减
14.2 限时订单概率修正
限时订单压力按：
剩余时间是否足够完成当前订单路线。
订单时间状态	判断方式	内部事件概率修正
时间充足	剩余时间明显大于预计路程	0
时间略紧	能完成，但余量不多	0.05
刚好够	基本没有容错	0.1
无法按原期限完成	预计会进入第一次超时，只能拿 50% 报酬	0.2
已进入无报酬风险	再拖就没有报酬，或已经确定拿不到报酬	0.3
说明：
订单无法按原期限完成，不是直接失败。
第一阶段是报酬降到 50%。
第二阶段才是没有报酬。
内部压力也按这两个阶段递增。

十七、距离安全点不直接加压
距离安全点远，不直接增加内部事件概率。
原因：
两个驿站之间本来就可能相隔 20–40 格。
如果距离远直接加压，会变成长期常驻惩罚。
所以：
项目	是否直接加内部事件概率
距离安全点远	不直接加
距离安全点远且补给不足	由补给动态压力体现
距离安全点远且车况差	由车况 / 零件压力体现
距离安全点远且士气低	由士气压力体现
距离安全点远且订单快超时	由订单动态压力体现
最终口径：
距离本身不是压力。
撑不到下一个安全点才是压力。

# 5.4 事件冷却系统

【原表内容】来源文件：`《余烬商队》第 5.4 类：事件冷却系统大报告 v1.1 修正版.xlsx`。以下尽量保留原文件内容、数值、特殊情况和原表例子；若与第 1 章最终覆盖口径冲突，以第 1 章为准。


《余烬商队》第 5.4 类：事件冷却系统大报告 v1.1 修正版
一、核心修正
第 5.4 类事件冷却系统不采用“强制禁止触发”的硬冷却。
所有事件冷却统一改为：
降权冷却。
也就是说，某类事件刚发生过之后，接下来一段步数内，该事件不会完全消失，而是权重下降。
下降幅度根据事件类型不同而不同。
这样可以避免两个问题：
玩家发现规律后，知道某几步绝对安全，产生机制割裂感；
野外风险变得过于机械，失去不可控的远行气质。
因此，5.4 的最终原则是：
冷却不等于禁止，冷却等于降权。
二、5.4 的系统定位
5.4 不重做事件权重。
5.2 已经决定外部事件基础权重。
5.3 已经决定内部事件压力概率。
5.4 只负责在事件抽取前增加一层节奏修正：
系统	负责内容
5.2 外部事件系统	决定不同区域、天气、路线下的外部事件基础权重
5.3 内部事件系统	决定商队压力如何影响内部事件概率
5.4 事件冷却系统	在事件刚发生后，短期降低同类事件权重，避免重复刷屏
5.4 的目标不是降低难度，而是控制节奏。
三、设计原则
3.1 所有事件都只做降权冷却
包括 E1 遇敌。
遇敌仍然需要很强的冷却保护，但不能完全禁止。
所以 E1 遇敌采用“极重降权冷却”。
也就是说：
刚发生过遇敌后，接下来几步内遇敌权重大幅下降，但仍保留极低概率。
这样既能避免连续战斗过密，又不会让玩家感觉系统在硬切状态。
3.2 冷却期间事件仍然保留在池中
冷却中事件不会被移出事件池，只是权重乘以一个保留比例。
例如：
原始权重	冷却保留比例	冷却后权重
20	0.15	3
10	0.3	3
8	0.4	3.2
这样事件仍然可能出现，只是概率显著降低。
3.3 遇敌降权要比其他事件更重
虽然不禁止遇敌，但遇敌的冷却降权必须明显。
原因是：
战斗是高消耗、高时间成本事件。
如果连续遇敌太频繁，游戏会变成刷怪。
但如果完全禁止遇敌，又会产生机械安全窗口。
所以最终折中为：
遇敌冷却期间保留 10%–15% 权重。
这代表：
不是绝对安全；
但连续遇敌概率很低；
灰烬区、危险区仍然保留压迫感；
玩家不会轻易摸出“固定安全步数”。
3.4 内部事件不设总冷却，只做同类型降权
内部事件仍然服从 5.3 的压力系统。
商队状态正常时，不会每天发生内部事件。
商队接近崩溃时，内部事件可以连续发生。
5.4 只限制同类型重复。
也就是说：
内部事件可以连续爆发，但不要连续刷同一个矛盾。

如果坏事来自玩家主动选择高风险选项，则 badEventStreak 计数减弱。
情况	处理
系统强制坏事	正常增加
玩家主动选择危险路线	不增加，严重后果时 +1
玩家明知有兽迹仍追踪	不增加，严重后果时 +1
玩家拒绝休整继续赶路	不增加，严重后果时 +1
玩家没有合理选择空间	正常增加
6.4 badEventStreak 权重修正
badEventStreak	处理
0	正常
1	正常
2	外部强负面事件权重额外降低 30%
3	外部强负面事件权重额外降低 60%，提高缓冲类事件权重
4+	外部强负面事件权重额外降低 80%，大幅提高情报、落脚点、地标、无事件权重
注意：
这里仍然不是禁止。
即使 badEventStreak 到 4+，负面事件也只是极重降权，不是完全消失。
七、冷却权重叠加规则
如果一个事件同时受到“大类冷却”和“连续坏事保护”，两个修正可以叠乘。
公式：
最终权重 = 基础权重 × 大类冷却保留比例 × badEventStreak 修正比例
表现【原表例子】
示例：
某区域 E1 遇敌基础权重为 20。
刚遇敌后，E1 进入 4 步冷却，保留 15%。
同时 badEventStreak = 3，外部强负面事件额外降低 60%，即保留 40%。
最终：
20 × 15% × 40% = 1.2
所以 E1 仍然可能出现，但概率非常低。
这比“完全禁止”更自然。

十、最终结论
第 5.4 类事件冷却系统最终采用“全降权冷却”方案。
核心口径如下：
《余烬商队》的事件冷却不使用强制禁止。
所有冷却都表现为权重下降。
遇敌、道路、资源、路线、遗迹、落脚点、关卡、地标等事件在发生后进入不同强度的降权期。
遇敌降权最重，但仍保留低概率出现，以避免玩家形成“固定安全步数”的机制感。
商队 / 旅人和情报 / 痕迹不设冷却，因为它们承担人烟、线索、铺垫和世界感功能。
内部事件不设总冷却，只设同类型降权。商队高压力时可以连续发生内部事件，但同一类矛盾短期内大幅降权，避免重复刷屏。
连续坏事保护也不禁止坏事，只进一步降低外部强负面事件权重，同时提高情报、落脚点、地标和无事件权重。
这套方案的目标不是让游戏变简单，而是让风险更像自然发生，而不是系统机械开关。

5.5 时间结算规则 v1.2：无歧义分类版
一、总原则
《余烬商队》的时间结算只使用整数天。
最终定稿：
玩家在地图上走 1 格 = 1 天。
事件要么不额外加天数，要么 +1 天 / +整数天。
不存在 0.5 天、半天、小数天。
二、时间结算的两个层级
时间结算必须分清两个层级：
层级	含义	是否固定发生
基础移动时间	玩家在地图上移动 1 格，固定消耗 1 天	固定发生
事件额外时间	玩家在事件中选择停留、绕路、搜索、修理、休整等，额外增加天数	只有选择或事件结果触发时才发生
因此，一个事件里写“+1 天”时，意思是：
在“走这一格本来已经消耗 1 天”的基础上，额外再增加 1 天。
例如：
行为	实际消耗
走 1 格，没有额外事件	1 天
走 1 格，遇到资源点，快速搜索	1 天
走 1 格，遇到资源点，停下仔细搜索	2 天：移动 1 天 + 额外搜索 1 天
走 1 格，遇到断路，选择绕路	2 天：移动 1 天 + 绕路 1 天
走 1 格，遇到严重塌方，选择大绕路	3 天：移动 1 天 + 绕路 2 天
三、时间结算分类总表
大类编号	大类名称	子类 / 行为	触发来源	额外天数	说明

六、最终口径
5.5 的时间结算必须使用分类写法，不能把不同层级的行为混在同一列。
最终口径：
时间结算分为“基础移动时间”和“事件额外时间”。
玩家每走 1 格固定消耗 1 天，这是基础移动时间。
事件额外时间只在玩家选择停留、绕路、深入搜索、修理、休整、撤退等情况下产生。
所有额外时间只能是 0 天、+1 天、+2 天或其他整数天。
每个事件表都必须标明时间分类、是否额外加天、额外天数、触发条件、订单是否扣时、补给是否消耗，并给出示例。
不允许出现 0.5 天、半天、小数天，也不允许只写“时间增加”而不说明增加多少天。
