# 文档优先级与修订规则

## 一、为什么要有优先级

随着设计推进，早期草案会被后续阶段报告覆盖。为了避免 Solo / Qoder / 开发工具误读旧口径，需要建立明确优先级。

## 二、优先级规则

1. 后写的正式阶段报告优先于早期草案。
2. `01_系统大类/` 下的阶段报告优先于根目录散文件。
3. 新版本明确废弃的旧口径，不再有效。
4. 同一系统内，修改日期越晚，越接近当前口径。
5. 如果新旧文件冲突，以新文件为准；旧文件应归档或标记为低优先级。
6. 不允许在 active README 中同时保留互相冲突的规则。

## 三、修订规则

以后每一版设计修订时：

1. 不要新建大量重复小文件；
2. 优先在当前系统目录下修订对应阶段报告；
3. 如果需要保留旧版，移动到归档目录，并标注"已被新版覆盖"；
4. README 只保留当前有效口径；
5. 废弃规则必须从 active 文档中删除，不要和新规则并列；
6. 重大修改要在文件开头增加"修订说明"。

## 四、命名规则

正式阶段报告使用：

第X.Y类_主题_v版本号.md

例如：

第6.2类_重伤记忆招募与商队扩展系统_v0.1.md
